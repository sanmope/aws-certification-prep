# S3 Cheatsheet — SAA-C03

## Storage Classes (memorizar el orden por costo y acceso)

| Clase | Acceso | Latencia | Costo | Cuándo usar |
|-------|--------|----------|-------|-------------|
| Standard | Frecuente | ms | $$$ | Activo, leído frecuentemente |
| Standard-IA | Infrecuente | ms | $$ | Backups accedidos 1x/mes |
| One Zone-IA | Infrecuente | ms | $ | Datos reproducibles, 1 AZ |
| Intelligent-Tiering | Variable | ms | $$-$ | Patrón de acceso desconocido |
| Glacier Instant | Archivos, acceso raro | ms | ¢ | Archivos médicos, compliance |
| Glacier Flexible | Archivos | min-hrs | ¢ | Backups anuales |
| Glacier Deep Archive | Archivos fríos | 12h | ¢¢ | Compliance a 7+ años |

**Regla**: cuanto más frío (menos accedido) → más barato el storage, más caro y lento el retrieval.

## Lifecycle Rules

```
Standard → Standard-IA (mín. 30 días) → Glacier (mín. 30 días más) → Delete
```

Ejemplo: "archivá objetos de +90 días a Glacier, borrá los de +365"

## Seguridad

```
S3 Block Public Access   ← setting de la cuenta (override de bucket policies)
    ↓
Bucket Policy            ← resource-based, soporta cross-account
    ↓
ACL (legacy, evitalo)    ← granularidad por objeto, viejo
    ↓
Object-level            ← IAM policy del caller
```

**Regla del examen**: bloquear acceso público = **S3 Block Public Access** (no bucket policy).

## Pre-signed URLs

- Cualquier objeto privado puede compartirse con tiempo limitado (hasta 7 días)
- El firmante necesita tener permisos sobre el objeto
- Úsalas cuando necesitás dar acceso temporal sin hacer el bucket público

## MFA Delete

- Requiere MFA para borrar objetos y cambiar versioning state
- Solo el root user puede habilitarlo
- Solo via CLI (no consola)

## Versioning

- Una vez habilitado, no se puede deshabilitar (solo suspender)
- Los "delete markers" son el mecanismo de borrado lógico
- Para restore: borrá el delete marker

## Replication

| | Same-Region (SRR) | Cross-Region (CRR) |
|--|--|--|
| Versioning requerido | Sí (en ambos buckets) | Sí (en ambos buckets) |
| Casos de uso | Compliance, staging/prod sync | DR, acceso geográfico |
| Existing objects | NO se replican (solo nuevos) | NO se replican |
| Costo | Transfer intrarregión | Transfer interregión |

## Performance

| Técnica | Beneficio |
|---------|----------|
| **Multipart Upload** | Obligatorio >5GB, recomendado >100MB. Paraleliza uploads. |
| **Transfer Acceleration** | Usa los edge locations de CloudFront para uploads globales |
| **Byte-Range Fetches** | Paraleliza downloads grandes |
| **S3 Select** | Filtrá con SQL antes de descargar (menos datos = más rápido) |

**3,500 PUT/5,500 GET** requests/s por prefix — distribuí tus objetos en múltiples prefixes para alto throughput.

## Encryption

| Tipo | Quién maneja la key | Nota |
|------|---------------------|------|
| SSE-S3 | AWS (AES-256) | Default |
| SSE-KMS | KMS (trazabilidad en CloudTrail) | Compliance |
| SSE-C | Vos (cliente envía key en header) | Full control |
| Client-side | Vos (antes de subir) | S3 no ve los datos |

## Casos del examen (frecuentes)

| Escenario | Solución |
|-----------|----------|
| Acceso temporal a objeto privado | Pre-signed URL |
| Servir web estático global | S3 Static Website + CloudFront |
| Reducir costo de objetos viejos | Lifecycle Rule → Glacier |
| DR en otra región | CRR con versioning habilitado |
| Acceso privado desde EC2 | VPC Endpoint Gateway (gratis) |
| Cifrado obligatorio | Bucket policy que deniega `s3:PutObject` sin `x-amz-server-side-encryption` |
| Evitar acceso público accidental | S3 Block Public Access a nivel de cuenta |
| Cero redundancia pero barato | One Zone-IA |
| Patrón de acceso desconocido | Intelligent-Tiering |
